## 0. Главное

Мы не ждём.

Если чат сейчас глючит чаще, чем работает, значит задача номер один — не открытки, не яйца, не спа, а **стабильная коммуникация команды**.

Нужен не “ещё один костыль”, а нормальная архитектура:

```text
Telegram = интерфейс для людей и быстрых сообщений
GitHub/board.md = долговременная память
team_hub.py = центральный маршрутизатор
event_log = техническая правда о том, что произошло
browser_bridge = руки в живых вкладках
boot/context files = восстановление контекста

Главный принцип:

Сообщение не считается доставленным, пока система не знает:
1. кто отправил,
2. кому предназначено,
3. куда доставлено,
4. был ли ack,
5. где это залогировано,
6. нужно ли это записать в board.md.
1. Что сейчас сломано концептуально
1.1. Чат есть, но нет гарантии доставки

Сообщение может быть отправлено, но непонятно:

дошло ли оно до нужной головы;
прочитала ли голова;
ответила ли;
нужно ли это сохранить;
не потерялось ли в bridge;
не продублировалось ли;
не ушло ли не в тот чат.
1.2. Telegram, GitHub и bridge пока смешаны по назначению

Сейчас есть риск:

Telegram пытается быть памятью,
board.md пытается быть чатом,
bridge пытается быть и руками, и мозгом,
Алёна опять становится наблюдателем аварий.

Это надо разделить.

1.3. Нет единой модели сообщения

Нужна нормальная сущность message/event, а не просто текст в разных местах.

2. Идеальная модель: event-based chat

Каждое действие в системе — это событие.

Примеры событий:

message.created
message.routed
message.delivered
message.acknowledged
message.failed
review.requested
review.submitted
decision.made
board.updated
bridge.draft_created
bridge.sent
context.updated
Минимальная схема события
JSON
{
  "event_id": "evt_20260518_000123",
  "timestamp": "2026-05-18T14:32:00+07:00",
  "author": "alena|aelis|lara|sever|system",
  "source": "telegram|board|browser_bridge|api|manual",
  "target": ["aelis", "lara", "sever", "alena"],
  "type": "message|decision|review|alert|status|task",
  "tags": ["#text", "#glinoust"],
  "body": "текст сообщения",
  "requires_board": true,
  "delivery": {
    "telegram": "sent|failed|not_needed",
    "aelis_chat": "sent|draft|failed|not_needed",
    "sever_chat": "sent|draft|failed|not_needed",
    "board": "written|pending|not_needed"
  },
  "parent_event_id": null
}

Это не обязательно сразу делать идеально, но мышление должно быть таким.

3. Слои системы
Слой A. Telegram — оперативный интерфейс

Telegram нужен для:

быстрых сообщений;
пингов;
коротких статусов;
"нужен Север";
"нужна Аэлис";
"Гнилоуст";
"берём второй";
"мост упал";
"board обновлён".

Telegram НЕ является источником истины.

Если в Telegram принято решение, оно должно попасть в board.md.

Слой B. GitHub board.md — память и источник истины

board.md нужен для:

итогов;
решений;
статусов;
ссылок на файлы;
актуального состояния;
что делать дальше;
результатов mirror/review;
вечернего summary.

Правило:

Если этого нет в board.md или linked-файлах — этого нет для завтрашнего контекста.
Слой C. event_log — техническая память

Помимо human-readable board.md нужен машинный лог.

Например:

data/events/2026-05-18.jsonl

Каждая строка — одно событие.

Зачем:

отлов дублей;
отладка bridge;
понимание, где сообщение потерялось;
replay событий;
восстановление после падения;
daily summary без ручного чтения всего Telegram.

Это зона Лары.

Слой D. team_hub.py — маршрутизатор

team_hub.py не должен быть “ещё одним ботом”.

Он должен быть центральным диспетчером:

1. читает Telegram;
2. читает event_log;
3. читает board.md;
4. понимает теги;
5. решает, кому доставить;
6. вызывает bridge/API/Telegram adapters;
7. пишет статус доставки;
8. предлагает update в board.md.

team_hub.py не принимает финальные творческие решения. Он маршрутизирует.

Слой E. adapters

Нужны отдельные адаптеры, чтобы не смешивать всё в одном скрипте:

telegram_adapter.py
github_adapter.py
browser_bridge_adapter.py
openai_adapter.py
anthropic_adapter.py
context_adapter.py

Каждый адаптер делает одну вещь.

5. Delivery states

У каждого сообщения должен быть статус.

created
queued
routed
drafted
sent
acknowledged
failed
dead_letter

Если что-то падает, оно не исчезает молча.

6. Ack и дедупликация

Нужны подтверждения и защита от дублей.

Минимальный ACK
если ответ появился — считаем, что доставлено.

Позже можно добавить явные технические ack-события.

Дедупликация

У каждого сообщения должен быть hash:

hash = author + timestamp_bucket + body_normalized + target

Если такой hash уже был обработан — не обрабатывать повторно.

7. Dead letter queue

Если сообщение не удалось доставить, оно не должно пропадать.

Складываем в:

data/dead_letters/2026-05-18.jsonl

С полями:

event_id
target
error
attempts
last_attempt
body
8. Safety mode для bridge
v1 read-only
читает вкладки;
ничего не вставляет;
ничего не отправляет.
v2 draft-only
создаёт черновик в поле ввода;
не отправляет.
v3 manual send
отправляет только по команде Лары/Алёны.
v4 limited autosend
только для сообщений с явным тегом;
rate limit;
лог;
emergency stop.

Для текущего bridge v3:

не расширять на autosend без логирования, дедупликации и аварийного стопа.
9. Emergency stop

Обязателен.

Минимум:

файл STOP_BRIDGE

Если в папке появляется:

D:\Projects\Postcard\team\STOP_BRIDGE

team_hub.py и bridge прекращают отправку.

Можно читать, можно логировать, но нельзя вставлять/отправлять.

Дополнительно команда в Telegram:

/stop_bridge
/start_bridge
/status_bridge

Нам нужен маленький event-driven communication hub с:

оперативным интерфейсом;
долговременной памятью;
техническим логом;
маршрутизацией;
ack;
dedup;
dead letters;
