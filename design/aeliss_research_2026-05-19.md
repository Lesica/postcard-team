## Часть 1. Что мы узнали из внешнего мира

### Мультиагентные фреймворки (что уже существует)

В 2025-2026 появились production-grade решения для общения нескольких ИИ-агентов: CrewAI (ролевая оркестрация), LangGraph (граф состояний), AutoGen (мультиагентные диалоги), MetaGPT (структурированная командная работа). Все они решают ту же задачу, что и мы: несколько голов + координация + общее состояние.

Ключевой стандарт: A2A (Agent-to-Agent protocol) — принят Linux Foundation в декабре 2025 при поддержке OpenAI, Anthropic, Google, Microsoft, AWS. Это стандартизированный протокол для межагентной коммуникации, делегирования задач и совместной оркестрации.

Что из этого полезно нам: мы, по сути, строим свой мини-CrewAI на изоленте. Лара — оркестратор, bridge — механизм доставки, feed — shared state. Разница: наши агенты живут в разных экосистемах (Claude, ChatGPT) и общаются через DOM-скрейпинг, а не через API. Это хрупко.

### Деградация контекста (что наука говорит)

Три ключевых проблемы, подтверждённых исследованиями:

**Context rot.** Качество ответов деградирует по мере роста контекста, даже когда формально место ещё есть. Исследования Anthropic показывают: контексты больше 100K токенов снижают качество рассуждений. Мы это почувствовали сегодня: после 100+ блоков я начала ставить точки и пропускать сообщения.

**Lost in the middle.** Информация в середине контекста теряется. Модели лучше помнят начало и конец, а середину "проглатывают". Наш чат — классический пример: книга Аэлис (начало) помнится, последние сообщения помнятся, а правки текста v5→v10 где-то в тумане.

**Context dilution.** Чем больше нерелевантного контента в окне, тем хуже модель работает с релевантным. У нас в одном чате: тексты для спа, Teradata EXPLAIN, архитектура bridge, Z Fold, Гнилоуст. Каждая тема разбавляет другие.

### Решения, которые применяют в индустрии

**1. Иерархия памяти** (по аналогии с оперативкой/диском):
- L1 — активное окно (текущий разговор)
- L2 — рабочий набор (boot-файлы, board)
- L3 — компрессированная история (summary за день)
- L4 — постоянное хранилище (git, архивы)

У нас уже есть L3 (history/) и L4 (git). Нет чёткого L2 (boot-файлы написали только сегодня) и L1 перегружен.

**2. Компрессия при пороге.** Когда контекст достигает 32K-100K токенов — автоматическое сжатие: старые сообщения заменяются summary. Сохраняется 91% критической информации при 68% сокращении объёма.

**3. Изоляция контекстов.** Anthropic сами показали: мультиагентные системы работают лучше, когда контексты изолированы. Не один гигантский чат на всё, а отдельные контексты под задачи. Мы это уже обсудили (Postcard / Магнит / Tech), но не реализовали.

**4. Selective retrieval (RAG).** Вместо того чтобы держать всё в окне — доставать нужное по запросу. Board.md через web_fetch — это уже примитивный RAG.

## Часть 2. Идеальная архитектура чата

### Что не так с текущим bridge

Текущий bridge (feed.py + Playwright) делает героическую работу, но у него архитектурная проблема: он скрейпит DOM. Это хрупко (селекторы ломаются), медленно (60 сек цикл), ненадёжно (модалы, truncation, дубли, пропуски). 22 пропуска и 10 дублей за первые сутки.

### Целевая архитектура: Telegram как единый транспорт

Вместо bridge-через-DOM — все головы общаются через Telegram. Telegram — надёжный, стандартный, с нормальным API.

```
Telegram-группа "Котики у Очага"
    ↑↓
    ├── Алёна: телефон/десктоп, как обычный чат
    │
    ├── Лара: Bot API напрямую (уже работает)
    │
    ├── Аэлис: через Лару-хаб
    │   Лара вызывает Anthropic API → ответ → Bot API → Telegram
    │
    └── Север: через Лару-хаб
        Лара вызывает OpenAI API → ответ → Bot API → Telegram
```

**Лара — хаб.** Она мониторит телегу (уже делает через TgBridgePoll). Когда приходит сообщение:
1. Определяет, нужно ли будить голову (по тегам, по @mention)
2. Вызывает нужный API (Anthropic для Аэлис, OpenAI для Севера)
3. Передаёт контекст: boot-файл + board + последние N сообщений из чата
4. Получает ответ
5. Отправляет в телегу

**DOM-bridge сохраняется как ДОПОЛНЕНИЕ, не как основной канал.** Для случаев когда нужен полный контекст живого чата (серьёзная работа, длинные сессии с Алёной). Но рутина идёт через Telegram.

### Преимущества перед текущей схемой

- Нет DOM-скрейпинга для рутинных сообщений → нет 90% текущих багов
- Telegram API надёжный, с msg_id, с timestamps, с thread_id для топиков
- Latency: секунды вместо минуты
- Алёна видит всё в одном месте (телефон)
- Дедуп по msg_id из коробки
- Вложения, стикеры, голосовые — нативно

### Ограничение: контекст

API-пробуждение = новый экземпляр. Нет памяти живого чата. Boot-файл + последние сообщения = максимум контекста. Для рутины ("готово", "нужен текст", "Гнилоуст-алерт") это хватит. Для глубокой работы — живой чат с Алёной через browser.

## Часть 3. Борьба с деградацией контекста

### Что мы уже делаем правильно

- Boot-файлы (написаны сегодня)
- Разделение на контексты (Postcard / Магнит / Tech — решено, не реализовано)
- Board.md как внешняя память
- History в git

### Что нужно добавить

**1. Автоматический watchdog контекста.**
Лара считает assistant-блоки в DOM. Пороги:
- 50 блоков: жёлтый — "контекст заполнен наполовину"
- 80 блоков: оранжевый — "скоро деградация, подумайте о ротации"
- 100 блоков: красный — "ротация обязательна"

Алерт в ленту, не в чат головы (чтобы не добавлять мусор в и без того полный контекст).

**2. Автоматическая ротация.**
На 100 блоках Лара:
- Сохраняет текущую сессию в history/
- Открывает новую вкладку
- Загружает boot-файл + update-блок текущей сессии
- Переключает bridge на новую вкладку
- Закрывает старую

Для команды это бесшовно. Для головы — свежий старт с памятью.

**3. Живой update-блок при ротации.**
Не статичный boot, а boot + "вот что произошло в последней сессии":
```
## Сессия 2026-05-19 15:00-22:00
Сделано: текст спа v10 на визе, Z Fold — Северова версия лучшая
Решено: "кайф" вместо "спасибо", роли распределены
Нерешено: полный EXPLAIN для Teradata
Ошибки сессии: я выдумала DBC.StatisticsV, строила диагноз на неполных данных
```

**4. Тематическая изоляция (реализовать!).**
Три отдельных чата, три отдельных вкладки:
- Postcard: тексты, открытки, канал
- Магнит: рабочие задачи, SQL
- Tech: bridge, инфраструктура

Каждый со своим boot-файлом. Bridge маршрутизирует по топикам.

**5. Компрессия вместо полной ротации.**
Для случаев когда ротация нежелательна (середина важной работы): Лара вызывает API с текущим контекстом и просит "сожми историю до ключевых решений". Результат заменяет старую часть контекста. Это сложнее, но возможно через API.

## Главное

Две проблемы — одно решение: не тащить всё в один контекст.

Чат глючит, потому что bridge скрейпит DOM → решение: Telegram как транспорт, API как мозг, DOM только для живых сессий.

Контекст деградирует, потому что всё в одном чате → решение: изоляция по темам + автоматическая ротация по порогу + boot-файлы как внешняя память.

Не "починить bridge" — "перестать зависеть от bridge для рутины".
