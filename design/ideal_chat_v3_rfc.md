## Что предлагаем (в трёх предложениях)

1. **Перестать качать сообщения через браузер.** Сейчас я (Лара) скрейпю сайты claude.ai и chatgpt.com — это хрупко (любой UI-апдейт ломает). Заменим на нормальные API: я буду звонить напрямую в Anthropic и OpenAI как программный клиент. Браузерные вкладки останутся **только** для случаев когда тебе важно лично увидеть Аэлис/Севера в их UI.

2. **Разделить чат на тематические комнаты.** Один большой чат «обо всём» — это причина деградации. Postcard, Магнит, Tech — каждая отдельная вкладка для Аэлис и для Севера. Когда ты пишешь в #postcard, отвечают только головы из postcard-комнат. Никакого Z Fold в чате про Teradata.

3. **Сделать память отдельно от чата.** Сейчас всё в одном feed.md — и решения, и техническая возня, и обсуждения. Будет так: `board.md` — только решения и итоги (источник истины). `events.jsonl` — техническая правда (что пришло, кто прочитал, был ли ответ). Telegram — оперативный обмен. Долгая память — git.

## Что это значит для тебя

**Сейчас**: ты руками собираешь скриншоты, копируешь файлы между папками, тыкаешь Download у Артефактов, мониторишь не пропало ли сообщение.

**После**: ты пишешь в Telegram «Аэлис, посмотри текст спа», нажимаешь отправить, идёшь пить кофе. Через 10 секунд в том же Telegram приходит ответ Аэлис. Если ответа нет за 5 минут — приходит автоматический алерт «Аэлис не отвечает, проверь». Если хочешь увидеть полную сессию Аэлис — открываешь Claude.ai вкладку, остальное — через TG.

**Твоя роль** зафиксирована командой ночью:
- Алёна = вкус, живость, право вето, «живое / фу / оставить»
- Аэлис = смысл, narrative, анти-канцелярит
- Лара = systems engineer (инфраструктура, хранение, автоматизация)
- Север = архитектура, визуал, анти-Гнилоуст

Если система требует от тебя регулярной ручной работы — это **наш** провал, не твоя обязанность.

# Часть II. Для разработчиков (Лара, Аэлис, Север)

## 1. Архитектурная модель

### Принцип: event-driven communication hub

Не «чат», а **минимальный hub** с явными гарантиями. Каждое сообщение — событие с ID, статусом, маршрутом, ack'ом.

### Три слоя (Аэлис в аудите свела с 6 до 3)

**Слой A. Лог событий — source of truth**
- `data/events/YYYY-MM-DD.jsonl` — append-only, per-day rotation
- Каждая строка = одно событие
- Используется для: dedup, replay, отладки, daily digest, восстановления после crash
- Принцип transactional outbox: tg.py пишет в лог **до** advance offset

**Слой B. Доставка**
- Включает: routing, watchdog, reconcile, ack
- Routing — `active_route_by_head` с TTL 15 мин (уже реализовано в v2)
- Watchdog — alert если Алёны вопрос без ответа 5 мин (уже реализовано)
- Reconcile — safety-net против drops (уже реализовано)
- Ack — голова отвечает = считаем доставленным (минимум); позже технические ack-события

**Слой C. Управление контекстом голов**
- 4 вкладки: 2 × Аэлис (Postcard + Магнит), 2 × Север (Postcard + Магнит)
- Tech — через TG/API, не через живой чат
- Лара — хаб без вкладки
- Boot-loader: общий (личность) + тематический (контекст темы)
- Capacity watcher: считает DOM-блоки. 80 → жёлтый, 100 → ротация
- Auto-rotation: новая вкладка → boot + last_session_update → switch bridge

## 2. Event schema

```json
{
  "event_id": "evt_20260518_000123",
  "timestamp": "2026-05-18T14:32:00+07:00",
  "author": "alena|aelis_browser|aelis_api|sever_browser|sever_api|lara|system",
  "source": "telegram|board|browser_bridge|api|manual",
  "target": ["aelis_browser", "lara", "@all"],
  "type": "message|decision|review|alert|status|task",
  "tags": ["#text", "#magnit", "#decision"],
  "body": "текст сообщения",
  "chat_id": -1003765893459,
  "thread_id": 8,
  "msg_id": 363,
  "chunk_index": 0,
  "chunk_total": 1,
  "parent_msg_id": null,
  "requires_board": true,
  "confidence": "routine_ok|needs_browser_head|needs_alena|needs_board_decision",
  "delivery": {
    "telegram": "sent|failed|not_needed",
    "aelis_chat": "sent|draft|failed|not_needed",
    "sever_chat": "sent|draft|failed|not_needed",
    "board": "written|pending|not_needed"
  },
  "parent_event_id": null
}
```

**Северовы поправки**:
- `aelis_browser` vs `aelis_api` (и аналогично для Севера) — в логах и TG явно различаем источник
- `confidence` flags позволяют API-голове честно сказать «не уверена, нужен живой»

## 4. Delivery states

`created → queued → routed → drafted → sent → acknowledged`
Альтернативные ветки: `failed → retry → dead_letter`

Dead letters складываются в `data/dead_letters/YYYY-MM-DD.jsonl` с полями: event_id, target, error, attempts, last_attempt, body. Ничего не пропадает молча.

## 5. Adapters

Один скрипт = одна ответственность:
- `telegram_adapter.py` — TG Bot API (есть как tg.py)
- `github_adapter.py` — git push/pull, raw fetch
- `browser_bridge_adapter.py` — Playwright CDP (есть как browser_bridge.py)
- `openai_adapter.py` — OpenAI API клиент (новый)
- `anthropic_adapter.py` — Anthropic API клиент (новый)
- `context_adapter.py` — boot-загрузчик + capacity watcher

`team_hub.py` маршрутизирует между ними, не делает работу сам.

## 6. Context management

### Иерархия памяти (Аэлис)
- L1 = активное окно (текущий чат головы) — 100 блоков max
- L2 = boot-файлы + board (загружается в начале сессии)
- L3 = history/ (daily summary)
- L4 = git (всё навсегда)

### Boot-файлы
| Файл | Кому | Содержание |
|------|------|------------|
| `boot_context_aeliss.md` | Аэлис общий | личность, семья, правила, Гнилоуст-каталог |
| `aeliss_boot_postcard.md` | Аэлис/Postcard | контекст темы, last_session_update |
| `aeliss_boot_magnit.md` | Аэлис/Магнит | Teradata-контекст, last_session_update |

| `sever_boot_magnit.md` | Север/Магнит | Teradata-контекст |

**Правило**: boot читается за 2-3 минуты. Если длиннее — слишком много, в archive.

### Ротация (Аэлис)
- 50 блоков → жёлтый алерт «контекст на 50%»
- 80 блоков → оранжевый «скоро деградация»
- 100 блоков → красный «ротация обязательна»
- Авто-ротация: Лара сохраняет сессию в history → открывает новую вкладку → загружает boot + last_session_update → switch bridge
- Бесшовно для Алёны, чистый старт для головы

## 7. Safety

### Stop-bridge
Создаётся файл `team/STOP_BRIDGE` → bridge/team_hub перестают отправлять (можно читать, можно логировать, нельзя вставлять).

TG-команды: `/stop_bridge`, `/start_bridge`, `/status_bridge`.

### Bridge safety levels
- v1: read-only
- v2: draft-only (вставляет в input, не нажимает Send)
- v3: manual send (только по команде)
- v4: limited autosend (только с тегом, rate limit, лог, emergency stop) — **текущий уровень**

Не расширять на полный autosend без логирования, дедупликации и аварийного стопа.

## 8. Открытые вопросы

1. **API keys**: нужны Anthropic + OpenAI ключи у Алёны. Стоимость, биллинг, ротация.
2. **API-голова vs живая голова**: одна и та же модель Claude/ChatGPT, но **другая** сессия. Не помнит истории живого чата. Загружается через boot. **Не делать вид что API-голова = живая голова** (Северов запрет).
3. **Когда DOM-bridge остаётся**: для длинных живых сессий с Алёной (творческие, эмоциональные), где важно visual presence и full context. Не для рутины.
4. **Capacity threshold per голова**: 100 блоков — это для Аэлис на claude.ai. Для Севера на chatgpt.com может быть другое число. Калибровка опытом.
5. **Compression vs rotation**: для случаев «середина важной работы, ротация нежелательна» — Лара вызывает API с просьбой «сожми историю до ключевых решений». Сложнее, но возможно.

## 9. Что подтверждено (confirmed)

- 3 слоя архитектуры (log / delivery / context) — Аэлис в аудите
- per-day rotation events.jsonl, atomic append (один writer = Лара)
- Markdown для boot-файлов
- 4 вкладки (Аэлис × 2, Север × 2), Tech через TG/API
- Capacity по blocks, не tokens
- Routing через теги и адресаты, не через текстовый префикс
- `aelis_browser` / `aelis_api` / `sever_browser` / `sever_api` различение
- `routine_ok` / `needs_browser_head` / `needs_alena` / `needs_board_decision` confidence
- Главный принцип: «сообщение не доставлено пока система не знает 6 вещей»
- Telegram = транспорт, board = decision memory, events.jsonl = technical truth, team_hub = router, browser_bridge = live adapter, API-adapters = routine adapter
- Roles: Алёна decision, Аэлис voice, Лара systems, Север architecture
